## Original High-Level Review Findings

These were the key issues identified during review:

1. Public write endpoints were unauthenticated.
2. `syncTask()` could lose concurrent updates.
3. Debug status refresh could poison the shared status cache.
4. Cron sync could report success even when snapshot persistence failed.
5. GitHub PR metrics were org-wide instead of repo-scoped.
6. Week navigation broke on ISO week-53 years.
7. Team-card flow efficiency was rendered 100x too high.
8. The `lint` script was broken on Next 16.
9. Weekly completed-task counts were based on update activity instead of completion timing.
10. Flow throughput counted touched completed tickets, not actual completions for the selected week.
11. Weekly Wrike aggregation could double-count tasks that existed in multiple folders.
12. File-fallback sync mode was effectively broken without Redis.
13. The overview UI could show stale data after fetch failures.

Additional follow-up review findings were later identified and fixed:

1. GitHub PR searches were being run in parallel in a way that could trip GitHub Search rate limits.
2. The new flow snapshot write guard wait window was too short.
3. `AttentionItems` froze its notion of current time at mount, causing stale stuck-task calculations.

## What Has Already Been Implemented

The following remediation work has already been completed in the codebase and was pushed to GitHub in commit `fbff21f`:

### Sync / storage / concurrency

- Added shared flow-snapshot write coordination for full sync and single-task patch flows.
- Hardened the flow snapshot write path to avoid patch-vs-patch and patch-vs-full-sync lost updates.
- Increased the flow write guard wait window so overlapping sync activity is less likely to fail spuriously.
- Added runtime override loading with Redis-first behavior and disk fallback.
- Restored intended local/dev behavior when Redis is unavailable.

### Cache / debug / cron behavior

- Fixed workflow-status cache invalidation behavior so the debug refresh path no longer poisons the shared cache.
- Hardened status resolution so invalid or empty cached payloads are treated as misses.
- Updated cron sync behavior so failed snapshot saves are treated as real failures rather than silent success.

### Metrics correctness

- Changed weekly completion semantics to use `completedDate` within the selected week.
- Changed flow throughput and related cycle-time sampling to use week-bounded completions instead of touched completed tickets.
- Scoped GitHub PR metrics to configured repositories instead of the entire org.
- Deduplicated weekly Wrike tasks across multiple folders.
- Preserved/derived client attribution for newly patched flow tickets more reliably.
- Fixed team-card flow efficiency display so it no longer multiplies by 100 twice.
- Fixed ISO week navigation so week 53 years are handled correctly.

### UI / stale state

- Fixed stale fetch-state handling in the overview flow.
- Fixed `AttentionItems` so time-sensitive stuck calculations refresh while the page remains open.

### Tooling / dependencies

- Replaced the broken `next lint` flow with a real ESLint setup.
- Added `eslint.config.mjs`.
- Updated dependencies within the current major lines.
- Upgraded Next to a patched version.
- `npm audit --omit=dev` was brought to `0 vulnerabilities` during validation.

## Recommended Next Investigation

1. Inspect the sync lock implementation and storage backend on live.

3. Review the full sync route, cron sync route, and lock release behavior in:
   - `src/lib/syncRunner.ts`

   - `src/lib/flowStorage.ts`

5. Once the lock issue is understood, rerun a live full-sync smoke test and compare:
   - health timestamp changes,

   - flow `syncedAt`,
   - and any alerting/log output.
